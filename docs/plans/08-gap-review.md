# 08. 구현 갭 리뷰

- 상태: 초안
- 작성일: 2026-03-25
- 기준: 현재 저장소 체크인 상태 (`apps/web`, `docs/plans`)

이 문서는 현재 코드베이스를 기준으로, 제품/설계 문서 대비 **부족한 부분**을 정리한 리뷰 문서입니다.
범위는 다음 5개 영역입니다.

- 디자인
- 프론트엔드
- 백엔드
- 웹 아키텍처/운영 품질
- 보안

---

## 0) 요약

현재 프로젝트는 **도메인 정의와 DB 스키마 설계는 비교적 잘 정리되어 있지만**, 실제 제품 구현은 아직 초기 단계입니다.

- 웹은 Next.js 엔트리와 헬스 체크 API 정도만 존재
- 모바일은 아직 구현 전
- 인증/인가, 관리자 권한, 입력 검증, 공용 UI 시스템, 운영 보안 계층은 대부분 미구현

즉, 지금 상태는 “서비스 운영 준비 단계”가 아니라 **설계가 앞서 있는 기반 공사 단계**로 보는 것이 정확합니다.

## 1) 우선순위 요약

| 우선순위 | 영역 | 이슈 | 영향 |
|---|---|---|---|
| P1 | 품질 / 개발 프로세스 | TDD 베이스라인 부재 | 도메인 규칙과 권한 경계가 코드로 고정되지 않음 |
| P1 | 백엔드 / 보안 | 인증/인가 계층 부재 | 사용자 분리, 관리자 보호, API 접근 통제가 불가능 |
| P1 | 백엔드 | 관리자 역할 모델 부재 | 웹 대시보드 접근 제어 설계와 실제 스키마가 충돌 |
| P1 | 백엔드 / 제품 | 상호 리뷰 요구사항과 DB 구조 불일치 | 리뷰 기능 구현 시 즉시 재설계 필요 |
| P2 | 백엔드 / 보안 | 입력 검증 및 도메인 제약 계층 부재 | 잘못된 데이터, 권한 우회, 상태 오염 가능성 증가 |
| P2 | 프론트 / 디자인 | 문서화된 디자인 시스템과 실제 UI 구현 분리 | 화면이 늘어날수록 일관성 붕괴 |
| P2 | 웹 / 보안 | Next.js 운영 보안 설정 부재 | 헤더/CSP/Referrer 정책 등 기본 웹 보안 미흡 |
| P2 | 프론트 | 실제 사용자 화면, 상태 화면, 오류 화면 미구현 | 제품 사용 자체가 아직 불가능 |
| P3 | 품질 | 테스트 부재 | 회귀와 권한/상태 전이 오류 탐지 어려움 |

## 2) 디자인

### 2.1 현재 상태

- `docs/plans/06-ui-theme.md`에 디자인 시스템 원칙과 토큰 규칙이 상세히 정의되어 있음
- 실제 웹 구현은 `app/page.tsx` 단일 랜딩 화면 수준
- Tailwind 확장 토큰은 일부 존재하지만, 문서 수준의 토큰 체계와 연결되어 있지 않음

### 2.2 부족한 부분

1. **디자인 토큰의 단일 소스가 없음**
   - 문서에서는 `packages/ui` 기준 토큰 관리 방식을 제안하지만 실제 구현은 없음
   - 현재 색상/그림자 값은 `tailwind.config.ts`와 페이지 유틸리티 클래스에 흩어져 있음

2. **문서와 실제 UI 톤이 다름**
   - 문서는 “Border > Shadow”, 무채색 중심, 의미 기반 컬러 사용을 강조함
   - 실제 화면은 그라디언트, blur, 큰 shadow 중심의 랜딩 스타일임

3. **타이포그래피 시스템 미정착**
   - Pretendard를 사용한다고 되어 있으나 실제 폰트 로딩이 없음
   - 숫자/가격 정렬, 상태 라벨, 보조 텍스트 계층도 아직 구현되지 않음

4. **도메인 UI 패턴 부재**
   - 예약 상태 배지, 가격 블록, 정책 메시지, 취소/환불 안내, 메시지 UI 같은 핵심 패턴이 실제 컴포넌트로 정리되어 있지 않음

### 2.3 권장 작업

1. `packages/ui` 또는 `apps/web/components`에 토큰과 기본 컴포넌트 레이어를 먼저 만든다.
2. 색상/간격/라운드/그림자 값을 raw utility가 아니라 semantic token으로 재정의한다.
3. `Button`, `Input`, `Card`, `Badge`, `FieldError`, `EmptyState`, `StatusChip`부터 만든다.
4. 폰트 로딩과 typography scale을 먼저 고정한다.

## 3) 프론트엔드

### 3.1 현재 상태

- Next.js App Router 기반 웹 진입점만 존재
- `/api/health` 외 실질 API 연결 화면 없음
- 모바일 앱은 문서만 있고 아직 구현 없음

### 3.2 부족한 부분

1. **실제 사용자 플로우가 없음**
   - 로그인
   - 회원가입/역할 선택
   - 요청자 예약 생성
   - 제공자 요청 수락
   - 예약 상세/채팅/리뷰
   - 관리자 대시보드

2. **라우트 구조가 설계만 있고 구현이 없음**
   - 문서에는 `/login`, `/admin`, `/admin/users`, `/admin/bookings` 등이 있으나 실제 파일은 없음

3. **상태 화면이 없음**
   - 로딩
   - 에러
   - empty state
   - not found
   - 권한 없음

4. **폼 아키텍처가 없음**
   - 예약 생성/프로필 수정/로그인 등 핵심 폼의 유효성 검사와 제출 흐름이 아직 없음

5. **접근성 기준이 반영되지 않음**
   - focus ring, 키보드 이동, aria, form error announcement, 명확한 heading 구조 검토가 필요

### 3.3 권장 작업

1. 웹에서 먼저 관리자 대시보드 MVP 또는 사용자 예약 플로우 중 하나를 선택해 수직 슬라이스로 구현한다.
2. 공용 레이아웃, 에러/빈 상태, 폼 필드 컴포넌트를 공통화한다.
3. 페이지 단위 구현보다 “도메인 플로우 단위”로 묶어 구현한다.
4. 페이지별 테스트 가능한 acceptance 기준을 만든다.

## 4) 백엔드

### 4.1 현재 상태

- Drizzle ORM 스키마는 상당 부분 작성되어 있음
- DB 연결 레이어 존재
- 서비스/액션/API 구현은 거의 없음

### 4.2 부족한 부분

1. **인증/인가 미구현**
   - Better Auth를 전제로 문서는 작성되어 있으나 실제 구현이 없음
   - 세션, OAuth, role 기반 접근 제어가 모두 비어 있음

2. **관리자 모델 부재**
   - 문서에는 Admin/CS/조회 전용 역할이 있으나 실제 `user_role` enum에는 `customer`, `cleaner`만 있음
   - 관리자 기능을 별도 테이블로 둘지, claim 기반으로 할지 결정 필요

3. **리뷰 모델 불일치**
   - PRD는 상호 리뷰를 말하지만 현재 `reviews.booking_id`는 unique
   - 예약당 리뷰 1개만 가능한 구조이므로 상호 리뷰 요구사항과 충돌

4. **도메인 검증 레이어 부재**
   - `rating` 범위 체크 없음
   - `cancelledBy`, `relatedType`은 자유 문자열
   - `services`, `tags`는 배열 원소 검증 계층이 아직 없음

5. **상태 전이 로직 미구현**
   - `pending -> confirmed -> in_progress -> completed` 전이를 누가 언제 바꾸는지 코드가 없음
   - 경쟁 수락 처리, 취소 정책, 완료 조건이 아직 구현되지 않음

6. **서비스 레이어와 트랜잭션 경계 미정**
   - 리뷰 작성 시 통계 업데이트
   - 예약 수락 시 cleaner 할당
   - 취소 시 알림 생성
   - 완료 시 리뷰 가능 상태 변경
   - 이런 작업이 단일 트랜잭션으로 묶여야 하는데 코드가 없음

7. **감사/운영 로그 계층 부재**
   - 관리자 액션, 상태 변경 이력, 로그인 시도, 잠금 정책 등의 운영 로그가 없음

### 4.3 권장 작업

1. 인증과 RBAC를 먼저 구현한다.
2. `admin` 모델을 확정하고 스키마/세션/미들웨어에 반영한다.
3. 리뷰 구조를 상호 리뷰 기준으로 재설계하거나 PRD를 단방향 리뷰로 수정한다.
4. `zod` 기반 validator와 DTO 계층을 도입한다.
5. 예약 상태 전이와 알림 생성을 서비스 레이어로 분리한다.

## 5) 웹 아키텍처 / 운영 품질

### 5.1 현재 상태

- Next.js App Router는 들어와 있으나 운영용 웹앱으로서의 기본 골격은 미완성

### 5.2 부족한 부분

1. **메타데이터 기본기 부족**
   - 타이틀/설명만 있고 Open Graph, Twitter card, canonical, robots 정책이 없음

2. **에러 경계 미구현**
   - `error.tsx`, `not-found.tsx`, `loading.tsx` 없음

3. **앱 구조가 페이지 1장에 머물러 있음**
   - 대시보드, 사용자 영역, 보호 라우트, 서버 액션 구조가 아직 없음

4. **운영 설정 부재**
   - 보안 헤더
   - 리다이렉트/리라이트 전략
   - 캐시 정책
   - 이미지 정책
   - 로그/모니터링 훅

5. **품질 게이트 부재**
   - 테스트 없음
   - 빌드 외 검증 기준 없음
   - PR 단위 체크리스트 없음

### 5.3 권장 작업

1. App Router 표준 파일 구조를 먼저 갖춘다.
2. 보호 라우트와 관리자 레이아웃을 분리한다.
3. `loading.tsx`, `error.tsx`, `not-found.tsx`를 도입한다.
4. 관측성 기준을 정한다.
   - request id
   - 에러 로깅
   - 관리자 액션 로그

## 6) 보안

### 6.1 현재 상태

보안은 “잘못 구현된 상태”라기보다, **구현해야 할 계층 자체가 아직 없는 상태**에 가깝습니다.

### 6.2 부족한 부분

1. **인증 없음**
   - 로그인, 세션, OAuth, 로그아웃 흐름 미구현

2. **인가 없음**
   - Customer / Cleaner / Admin 접근 제어 미구현
   - 본인 소유 예약/메시지/리뷰에 대한 접근 제한 코드 없음

3. **보안 헤더 없음**
   - CSP
   - `X-Frame-Options`
   - `Referrer-Policy`
   - `X-Content-Type-Options`
   - `Permissions-Policy`

4. **입력 검증 없음**
   - API payload 검증
   - enum 멤버십 검증
   - 폼 입력 정규화
   - 파일 업로드 검증 전략 없음

5. **공격 방어 전략 부재**
   - rate limiting
   - brute-force 방지
   - 계정 잠금
   - CSRF 전략
   - 관리자 보호 정책

6. **비밀값/환경변수 운영 규칙 부재**
   - `.env` 정책
   - provider secret 관리
   - 배포 환경 분리 규칙

7. **감사 가능성 부족**
   - 누가 언제 예약 상태를 바꿨는지
   - 누가 관리자 작업을 했는지
   - 로그인 실패가 몇 번 있었는지
   - 현재 저장 구조로는 운영 감사가 어려움

### 6.3 권장 작업

1. Better Auth 도입과 세션 전략 확정
2. RBAC 미들웨어 도입
3. 주요 API에 validator 적용
4. 보안 헤더와 CSP 기본값 설정
5. 관리자 영역 rate limit 및 감사 로그 설계

## 7) 권장 실행 순서

### Phase 0: TDD 베이스라인 구축

1. 테스트 도구 선정과 기본 설정
2. 첫 도메인 테스트 대상 선정
3. 기능 구현 전 failing test 작성 원칙 확정
4. `test` 스크립트를 실제 테스트 실행 기준으로 교체

### Phase 1: 서비스 경계 확정

1. 인증 방식 확정
2. 관리자 권한 모델 확정
3. 리뷰 모델 확정
4. 예약 상태 전이 규칙 확정

### Phase 2: 백엔드 안전장치 구축

1. auth + session
2. RBAC
3. validator/DTO
4. service layer + transaction
5. 감사 로그

### Phase 3: 프론트 MVP 구축

1. 공용 UI 토큰/컴포넌트
2. 로그인/보호 라우트
3. 관리자 대시보드 기본 골격
4. 예약 목록/상세/상태 변경 화면

### Phase 4: 운영 품질 강화

1. 보안 헤더
2. 에러/로딩/빈 상태
3. 테스트
4. 로깅/모니터링

## 8) 결론

이 프로젝트의 강점은 **도메인이 좁고 명확하며, DB 스키마 설계가 이미 상당히 구체적**하다는 점입니다.
반면 현재 가장 큰 리스크는 “화면이 부족하다”가 아니라, **서비스 경계(auth/RBAC/관리자/리뷰 모델)가 아직 코드로 고정되지 않았다**는 점입니다.

따라서 다음 구현은 디자인 디테일보다 먼저 아래 순서로 진행하는 것이 안전합니다.

1. 인증/인가
2. 관리자 모델
3. 리뷰/상태 전이 구조
4. 입력 검증
5. 공용 UI와 화면 구현
